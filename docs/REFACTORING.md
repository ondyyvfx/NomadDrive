# Доработка и рефакторинг проекта NomadDrive

> Документ описывает этап доработки платформы: переход от прототипа с
> демонстрационными (мок) данными к рабочей версии, в которой все элементы
> интерфейса подключены к реальной базе данных и функционируют. Может
> использоваться как отдельный раздел дипломной работы.

---

## 1. Постановка проблемы

Первоначальная версия интерфейса была собрана на основе готового дизайн-макета.
Визуально страницы выглядели завершёнными, однако значительная часть элементов
была **нефункциональной** или содержала **жёстко «зашитые» в код демо-данные**:

| Проблема | Где проявлялась | Последствие |
|---|---|---|
| Мок-данные в коде | Главная страница (массивы `RENT_CARS`, `BUY_CARS`, `PARTS`) | Карточки не отражали реальный каталог, клики ни к чему не вели |
| Некорректные данные в БД | Supabase: марка не совпадала с фотографией | «BMW X5» с фотографией Porsche, неработающие изображения |
| Битые ссылки | Похожие авто, редирект бронирования (`/cars/...`) | Переход на несуществующий маршрут (ошибка 404) |
| «Мёртвые» кнопки | Поиск, продажа, соцсети, контактная форма | Кнопка есть, но ничего не делает |
| Несуществующая страница | Ссылка «Забыли пароль?» (`/forgot-password`) | Ошибка 404 |
| Декоративный мусор | Случайные изображения (горы, посторонние объекты) в карточках | Несоответствие смысла блока и картинки |

Цель этапа — устранить разрыв между «выглядит готовым» и «работает»,
доведя каждую деталь до состояния, характерного для коммерческого продукта.

## 2. Очистка и пересоздание базы данных

### 2.1 Подход

Поскольку данные в облачной БД Supabase были несогласованными (неверные
соответствия «марка ↔ фотография», нерабочие ссылки на изображения),
было принято решение о **полном сбросе и пересоздании** данных с помощью
единого SQL-скрипта — `db/dump.sql`.

Скрипт **идемпотентный** (можно запускать повторно) и выполняет:

1. `DROP ... CASCADE` всех таблиц — удаление старой несогласованной структуры.
2. `CREATE TABLE` — пересоздание схемы из 9 таблиц.
3. Включение Row Level Security и настройку политик доступа.
4. Создание триггеров и функций.
5. Вставку выверенных демонстрационных данных.

### 2.2 Структура (9 таблиц)

`profiles`, `cars_for_rent`, `cars_for_sale`, `parts`, `promo_codes`,
`bookings`, `orders`, `order_items`, `reviews`.

Все типы и ограничения (`CHECK`, внешние ключи, индексы) приведены в
соответствие с TypeScript-типами приложения (`src/types/index.ts`), что
исключает рассогласование между БД и кодом.

### 2.3 Безопасность на уровне строк (RLS)

Настроены политики, реализующие принцип минимальных привилегий:

| Таблица | Чтение | Запись |
|---|---|---|
| Каталоги (`cars_for_rent`, `cars_for_sale`, `parts`) | Все | Только администратор |
| `bookings`, `orders`, `order_items` | Владелец или администратор | Владелец |
| `promo_codes` | Все (для проверки кода) | Только администратор |
| `profiles` | Все | Только владелец / администратор |

Для проверки роли используется функция `is_admin()` с атрибутом
`security definer` — она обходит RLS при чтении таблицы `profiles`, что
**предотвращает бесконечную рекурсию политик** (классическая ошибка при
настройке RLS в PostgreSQL).

### 2.4 Автоматизация

- **Триггер `on_auth_user_created`** — при регистрации пользователя в
  `auth.users` автоматически создаётся запись в `public.profiles`.
- **Триггер `trg_profiles_touch`** — поддерживает актуальность поля
  `updated_at` при изменении профиля.

### 2.5 Демонстрационные данные

| Сущность | Количество |
|---|---|
| Автомобили в аренду | 10 |
| Автомобили на продажу | 10 |
| Запчасти | 14 |
| Промокоды | 3 |

Цены указаны в тенге и приближены к рыночным; локации — реальные районы
Алматы и Астаны с координатами.

### 2.6 Методика выверки фотографий

Главное требование заказчика — **изображение должно соответствовать марке и
модели**, а ссылка должна гарантированно открываться. Применённая методика:

1. **Проверка доступности.** Каждая ссылка на изображение проверялась
   автоматически на код ответа HTTP 200 (нерабочие отбрасывались).
2. **Визуальная верификация.** Все кандидаты загружались и **просматривались
   вручную** — сверялось фактическое содержимое фотографии с подписью.

Такой контроль выявил грубые ошибки исходного набора: изображение с подписью
«кованые диски» в действительности оказалось пейзажем гор, среди «запчастей»
встречались посторонние объекты, фотография «Land Rover» содержала Tesla, а
«Mercedes G63» — Ford Mustang. В итоговом наборе подобные несоответствия
исключены полностью.

### 2.7 Применение

```text
Supabase Dashboard → SQL Editor → New query → вставить db/dump.sql → Run
```

Назначение администратора (после регистрации на сайте):

```sql
update public.profiles set role = 'admin'
where id = (select id from auth.users where email = 'ВАШ_EMAIL');
```

## 3. Подключение главной страницы к реальным данным

### 3.1 Было

Лендинг (`LandingPage.tsx`) — клиентский компонент, в который были «зашиты»
статические массивы автомобилей и запчастей с фиксированными изображениями.
Каталог на главной не имел отношения к содержимому БД, карточки не вели на
детальные страницы.

### 3.2 Стало

Применён архитектурный паттерн **Server Component → Client Component**:

```
page.tsx (Server Component)              LandingPage.tsx (Client Component)
  ├─ запрос featured rent/sale/parts      ├─ принимает данные пропсами
  │   из Supabase (limit 6/6/4)           ├─ маппинг строк БД → карточки
  └─ передаёт данные пропсами  ─────────► └─ карточки ведут на /rent/[id],
                                              /sale/[id], /parts
```

- Серверный компонент `page.tsx` параллельно (`Promise.all`) запрашивает
  доступные авто (`status = 'available'`) и свежие запчасти.
- Жёстко заданные массивы `RENT_CARS`, `BUY_CARS`, `PARTS` **удалены**.
- Введены типобезопасные интерфейсы (`RentCarRow`, `SaleCarRow`, `PartRow`) и
  функции нормализации (`rentToView`, `saleToView`), переводящие строки БД в
  единое представление карточки.
- Каждая карточка стала ссылкой (`next/link`) на детальную страницу товара.
- Добавлены пустые состояния («Скоро здесь появятся автомобили») на случай
  отсутствия данных.

## 4. Оживление интерфейса

### 4.1 Удаление нефункциональных элементов

- Удалены **иконки социальных сетей** (в hero-секции и блоке контактов) —
  они вели на заглушку `#`. В блоке контактов вместо них размещены реальные
  телефон, email и адрес.
- Удалены **посторонние декоративные изображения** из карточек блока
  «Почему выбирают нас»; для единообразия карточки получили стандартный
  элемент-стрелку.

### 4.2 Подключение кнопок

| Элемент | Было | Стало |
|---|---|---|
| Кнопка поиска (hero) | Без действия | Переход в каталог аренды `/rent` |
| Кнопки разделов | Без действия | `/rent`, `/sale`, `/parts`, `#contact` |
| «Получить оценку» (продажа) | Без действия | Переход к форме контакта `#contact` |
| «Читать манифест» | Без действия | Переход к разделу «Как это работает» |
| Аватары в отзывах | Сторонние фото-заглушки | Инициалы в фирменном стиле |

## 5. Исправление ошибок и отказов

| № | Дефект | Решение |
|---|---|---|
| 1 | Ссылки на похожие авто и редирект бронирования вели на `/cars/...` (404) | Исправлены на корректный маршрут `/rent/...` |
| 2 | Ссылка «Забыли пароль?» вела на несуществующую страницу | Создана рабочая страница `/forgot-password` (отправка письма через Supabase Auth) и `/reset-password` (установка нового пароля) |
| 3 | После входа игнорировался параметр `?redirect=` | Вход возвращает пользователя на запрошенную страницу |
| 4 | Кнопки «Связаться с продавцом» и «Чат» (продажа) не работали | Заменены на рабочие ссылки `tel:` и WhatsApp с предзаполненным сообщением |
| 5 | Несогласованный год в подвале (2025/2026) | Приведён к единому значению |

### 5.1 Восстановленный сценарий сброса пароля

```
/forgot-password ──(email)──► письмо Supabase ──(ссылка)──►
        /reset-password ──(новый пароль)──► /dashboard
```

Страница `/reset-password` подписывается на событие `PASSWORD_RECOVERY`
клиента Supabase, получает сессию восстановления из ссылки в письме и
позволяет задать новый пароль через `updateUser`.

## 7. Итоги

- Устранён разрыв между визуально готовым макетом и реальной
  функциональностью: каждый интерактивный элемент выполняет осмысленное
  действие.
- База данных приведена в согласованное состояние; данные и фотографии
  выверены, схема и политики безопасности воспроизводимы одним скриптом.
- Главная страница перестала быть статичной витриной и отражает актуальное
  содержимое каталога.
- Восстановлены критичные пользовательские сценарии (сброс пароля, переходы
  по каталогу, связь с продавцом).

### Известные упрощения (направления развития)

- Поля «город / даты / класс» в строке поиска главной страницы носят
  оформительский характер; кнопка поиска ведёт в каталог. Полноценная
  передача фильтров через параметры URL — отдельная задача.
- Контактные реквизиты (`+7 700 700 70 00`, `info@nomaddrive.kz`) являются
  заглушками и подлежат замене на реальные.
- Форма обратной связи на главной показывает подтверждение отправки, но не
  сохраняет заявку (отсутствует таблица лидов) — возможное расширение.
